관계대수와 관계해석
관계대수: 데이터베이스에서 원하는 정보와 그 정보를 검색하기 위해 명시하는 SQL의 이론적 기반 절차적 언어
주요 연산자(단항 연산자, 이항 연산자, 조인 연산자)
단항 연산자
Selection: 조건을 만족하는 튜플 선택
Projection: 특정 속성만 추출
Rename: 릴레이션의 이름 변경
이항 연산자
Union: 두 릴레이션에서 모든 튜플을 중복 없이 결합
Difference: 첫 릴레이션에만 있는 튜플 선택
Cartesian: 모든 튜플 조합을 생성
조인 연산자
Join: 두 릴레이션을 공통 속성 기준으로 병합

관계해석: 관계형 데이터베이스에서 연상을 표현하기 위한 수학적 논리 기반의 비절차적 언어
튜플 관계해석
도메인 관계해석

정리하면 관계대수는 SQL의 실행 기반, 관계해석은 SQL의 표현 기반, SQL은 이를 결합한 실용 언어이다.

트랜잭션: 하나의 논리적 기능을 수행하기 위한 작업의 단위이며, 하나의 트랜잭션은 모두 성공하거나 실패해야 함 -> 데이터베이스 내의 작업의 가장 작은 단위
특성(원자성, 일관성, 독립성, 영속성)
Atomicity(원자성): 트랜잭션 내 연산은 모두 수행되거나, 하나라도 실패하면 전부 취소되어야 함
Consistency(일관성): 트랜잭션 수행 전과 수행 후에도 데이터베이스는 항상 일관된 상태를 유지해야 함
Isolation(독립성): 동시에 실행되는 트랜잭션들은 서로 영향을 주지 않는 것처럼 실행
Durability(영속성): 성공적으로 완료한 트랜잭션의 결과는 시스템에 장애가 발생해도 영구적으로 반영

트랜잭션의 연산
Commit: 트랜잭션에서 수행한 변경 내용을 데이터베이스에 영구적으로 반영
Rollback: 트랜잭션 수행 중 발생한 변경 내용을 모두 취소하고 이전 상태로 복구

트랜잭션의 상태
활성화(Active): 트랜잭션이 실행 시작하여 SQL문 수행중인 상태
부분 완료(Partially Committed): Commit 요청이 들어온 직후의 상태 아직 DB에는 반영되지 않은 상태
완료(Committed): 트랜잭션이 Commit 수행되어 DB에 반영
실패(Failed): 오류가 발생하여 진행할 수 없는 상태
철회(Aborted): 실패한 트랜잭션이 Rollback되어 복구된 상태
다수의 사용자가 동시에 데이터베이스를 이용할 수 있도록 트랜잭션들이 병렬로 처리되어야 한다

병렬 수행의 문제점: 여러 트랜잭션이 동시 실행될 경우 처리 속도는 향샹되지만 이상현상 발생 가능
갱신 손실(Lost Update): 두 트랜잭션이 같은 데이터를 동시에 수행할 때, 한 트랜잭션의 결과가 다른 트랜잭션에 의해 사라지는 현상
더티 리드(Dirty Read): 아직 commit 되지 않은 데이터를 다른 트랜잭션이 읽는 현상
반복 불가능 읽기(Non-repeatable Read): 같은 데이터를 두 번 조회했는데 결과가 달라지는 현상
팬텀 리드(Phantom Read): 같은 조건으로 조회했는데 조회되는 행의 개수가 달라지는 현상

병행제어 기법: 병행 수행 환경에서 데이터의 일관성과 고립성을 유지하기 위한 제어 방법
락 기법: 데이터 사용 중일 때 다른 트랜잭션의 접근을 제한
-	Shared Lock: 읽기 허용 / Exclusive Lock: 읽기, 쓰기 모두 차단
타인스탬프 기법: 트랜잭션에 시간 순서 부여하여 충돌 시 먼저 시작한 트랜잭션 우선 처리
낙관적 기법: 충돌이 거의 없다고 가정하고 작업 완료 시점에만 검증 수행, 검증 시 이상 발생하면 Rollback
다중 버전 제어: 데이터의 여러 버전을 관리하여 읽기와 쓰기를 동시에 허용

트랜잭션 격리 수준(Isolation Level): 트랜잭션간 간섭을 어느 정도까지 허용할 것인지에 대한 기준
READ UNCOMMITTED: Commit 되지 않은 데이터 읽기 허용 – 거의 병행제어 하지 않음
READ COMMITTED: Commit 된 데이터만 읽기 허용 – Shared Lock 사용
REPEATABLE READ: 같은 데이터 반복 조회 시 결과 보장 – 읽기/쓰기 잠금 유지
SERIALIZABLE: 가장 높은 격리 수준으로 모든 병행 문제 방지 – 범위 잠금, 다중 버전 제어 

병행제어와 격리 수준으로 문제를 줄이더라도, 장애는 언제든지 발생할 수 있음

장애와 회복
데이터베이스의 장애
트랜잭션 장애: 트랜잭션 수행 중 오류 발생으로 정상적 수행 불가 상태
시스템 장애: 하드웨어 결함으로 정상적 수행 불가 상태 -> Log
미디어 장애: 디스크 장치 결함으로 디스크 저장된 데이터베이스 내용 손상 -> BackUp

회복 기법
회복 연산
Redo: 트랜잭션의 작업을 다시 실행하여 데이터베이스에 반영
Undo: 실패한 트랜잭션의 작업을 취소하여 이전 상태로 되돌림
지연 갱신 DB: commit이 수행될 때 한번에 데이터 변경
즉시 갱신 DB: 데이터 변경 트랜잭션이 수행될 때 즉시 데이터 변경 
로그 회복 기법: 트랜잭션 작업 내용을 로그에 기록하여 장애 발생 시 데이터베이스를 복구하는 기법
체크포인트 기법: 일정 시점의 데이터베이스 상태를 저장하여 장애 발생 시 복구 시간을 단축하는 기법
그림자 페이징 기법: 트랜잭션 수행 시 두 개의 페이지 테이블을 관리하며, 장애 발생 시 그림자 페이지로 복구하는 기법

분산 데이터베이스와 보안
분산 데이터베이스: 여러 물리적인 데이터베이스를 네트워크로 연결하여 하나의 논리적 데이터베이스로 운영하는 시스템
분산처리기: 자체적으로 처리 능력을 가지며, 지리적으로 분산되어 있는 컴퓨터 시스템
분산 DB: 지역적으로 분산되어 있는 데이터베이스로서 해당 지역에 맞게 구성
통신 네트워크: 분산 처리기들을 논리적으로 하나의 시스템처럼 동작할 수 있도록 하는 역할
장점: 지역 자치성이 높음, 분산 제어가 가능, 높은 효용성과 융통성, 융통성 있는 시스템 용량 확장성, 자료의 공유성이 향상, 시스템 성능의 향상, 높은 신뢰성과 가용성
단점: 복잡한 DBMS의 수행 기능, 소프트웨어 개발 비용의 증가, 불규칙한 응답 속도, 데이터베이스의 설계가 어려움, 잠재적 오류의 증가

분산 데이터베이스의 4대 투명성
위치 투명성: 사용하는 데이터베이스의 실제 위치를 알 필요가 없으며, 논리적인 명칭만으로 엑세스가 가능
중복 투명성: 동일 데이터가 여러 곳에 중복되어 있어도 사용자는 하나의 데이터가 존재하는 것처럼 사용
병행 투명성: 다수의 트랜잭션이 동시에 수행되어도 서로 영향을 받지 않음
장애 투명성: 장애에도 불구하고 트랜잭션을 정확하게 수행

데이터베이스 보안: 권한이 없는 사용자의 접근을 차단하고, 데이터의 기밀성, 무결성, 가용성을 유지하기 위한 조치
보안의 3대 요소
기밀성: 오직 허락된 사용자만이 데이터에 접근 가능
무결성: 권한이 있는 사용자만 수정할 수 있으며, 항상 일관되고 정확한 상태 유지
가용성; 허락된 사용자가 서비스를 필요로 할 때 지체 없이 데이터에 접근 허용
사용자 권한 관리(DCL)
GRANT: 사용자에게 특정 데이터베이스 객체에 대한 작업 권한을 부여
REVOKE: 이미 부여된 권한을 회수
접근 제어 모델
임의적 접근 제어(DAC): 데이터 소유자가 권한을 직접 부여하고 관리
강제적 접근 제어(MAC): 시스템 중앙에서 보안 등급에 따라 접근 권한을 규제
역할 기반 접근 제어(RBAC): 역할에 권한을 할당하고 역할을 배정하는 방식
보안 강화 기법
암호화: 데이터 전송, 저장 시 암호화를 적용하여 유출 피해 최소화
감사: 사용자 활동 로그를 기록하여 이상 징후 추적하고 사후 책임을 규명
SQL 인젝션 방어: 악의적인 SQL문이 실행되지 않도록 방어 체계를 구축
