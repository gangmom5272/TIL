데이터베이스 개론

데이터: 가공되지 않은 그대로의 객관적인 사실
정보: 목적에 따라 데이터를 처리하여 의미를 부여한 것, 의사 결정 문제에 대한 해결의 수단으로 사용

데이터와 정보 개념 정립의 중요한 이유
정확한 모델링 – 데이터를 정보로 바꾸는 구조를 이해해야 현실 세계를 데이터베이스로 정확히 표현
데이터 처리의 효율성 확보 – 어떤 데이터가 가치 있는 정보가 될 수 있는지 판단하여, 불필요한 데이터 제거 및 정확한 설계가 가능
의사결정 지원 – 정보는 의사결정을 하기 위한 기반으로서, 정보를 생성하는 데이터를 이해해야 정확한 결과 도출 가능
보안과 무결성 확보 – 어떤 데이터가 중요한 정보로 쓰이는지 구분하면, 보호해야 할 데이터와 그렇지 않은 데이터를 구분 가능

데이터 분류
정형 데이터: 구조가 정해진 데이터(엑셀 스프레드 시트, CSV 파일, RDBMS) -> 고정된 형태로 저장
반정형 데이터: 내부 정보를 표현하는 메타데이터가 포함(XML, JSON, 웹 로그 파일)
비정형 데이터: 구조화가 되어 있지 않은 데이터(영상, 이미지, SNS 게시글, 텍스트 문서) -> 고정된 형태로 표현 불가

데이터베이스의 개념
특정 목적을 위해 관련 있는 데이터들을 구조적으로 모아 놓은 저장소
질의처리기(Query Processor): 사용자의 질의(SQL)를 처리하여 데이터베이스에서 원하는 정보를 추출하는 역할
트랜잭션 관리자: 트랜잭션의 시작, 종료, 복원 등의 작업을 제어하여 데이터 일관성을 유지하는 역할
	트랜잭션: 데이터베이스의 논리적인 기본 처리 단위
저장 데이터 관리자: 데이터베이스의 물리적 공간에 대한 관리를 통한 데이터 효율적 저장을 지원하는 역할

데이터 관점의 특성
공용 데이터: 여러 사용자가 다른 목적으로 데이터를 공동 이용
통합된 데이터: 동일한 데이터가 중복되어 있지 않음
저장된 데이터: 저장매체에 데이터를 저장
변화되는 데이터: 새로운 데이터 추가, 수정, 삭제에도 현재의 정확한 데이터 유지

시스템 운영적 특성
실시간 접근성: 실시간 처리에 의한 응답이 가능
계속되는 변화: 데이터베이스의 상태는 계속적으로 변화
동시 공용: 다수의 사용자가 동시에 같은 내용의 데이터 이용 가능
내용에 의한 참조: 사용자가 요구하는 데이터 내용으로 데이터를 탐색

데이터베이스관리시스템(DBMS) 
등장배경: 파일시스템의 중복을 없애기 위해서
사용자와 데이터베이스 사이에서 데이터베이스를 관리해주는 프로그램

특징
구조화된 저장
테이블 구조를 활용하여 중복을 최소화하고, 일관성과 무결성 유지 가능
제약조건 설정
데이터의 일관성과 무결성 유지를 위하여 제약조건 설정 가능
동시 접근제어
여러 사용자의 동시 접근을 트랜잭션 단위로 관리하여 효율적으로 관리
보안성
사용자 인증 및 권한 부여를 활용한 접근 제어 및 암호화 지원, 로그 파일 활용 복구 가능

필수 기능
정의 기능: 데이터의 타입, 구조, 이용 방식, 제약 조건 등을 정의
조작 기능: 데이터의 검색, 삽입, 삭제, 갱신 등을 처리
제어 기능: 데이터 무결성 유지, 정당한 사용자 허용 위한 권한 검사, 병행 제어

DBMS의 종류
계층형 DBMS: 트리구조로, 부모-자식 관계를 기반으로 데이터를 저장(빠른 접근 속도, 데이터 중복 가능성)
네트워크 DBMS: 네트워크 상의 그래프 형태로 표현한 구조(유연하지만 설계 및 복잡한 유지 관리)
객체지향 DBMS: 객체지향 프로그래밍의 객체 개념을 데이터베이스에 적용(객체 간 관계 표현에 적합)
관계형 DBMS: 데이터를 테이블(행과 열)형태로 저장(가장 널리 사용되는 DBMS, 정형 데이터에 적합)
NoSQL: 다양한 데이터를 처리가 가능(대용량 데이터, 비정형 데이터 처리에 적합하며, 높은 확장성)

데이터베이스 언어
데이터베이스를 구축하고 이용하기 위한 데이터베이스 시스템과의 통신 수단
정의어(DDL – Data Definition Language): 데이터베이스의 구조를 정의하는 언어
조작어(DML – Data Manipulation Language): 데이터를 조회하거나 삽입, 수정, 삭제하는 언어 
제어어(DCL – Data Control Language): 데이터 무결성 유지, 보안 및 권한 제어, 회복과 병행제어를 수행하는 언어

컴퓨터 언어는 문제를 해결하는 방식에 따라, 비절차적 언어와 절차적 언어로 나눌 수 있다.
비절차적 언어: 무엇을 할 것인지만 명시하고, 어떻게 수행할지는 시스템이 처리하는 언어
	SQL – 사용자는 데이터가 어떻게 처리되는지는 지정하지 않음, 어떤 데이터를 원한다는 선언만 수행, DBMS내부에서 최적 실행 계획을 생성하여 처리
절차적 언어: 무엇을 할 뿐만 아니라, 어떻게 할 것인지까지 직접 명시하는 언어
	PLSQL, T-SQL – SQL 문장을 절차적 흐름 속에서 사용, 변수 선언, 조건문, 반복문 등 포함, 복잡한 로직이나 배치 처리를 작성할 때 매우 유용

데이터베이스의 스키마
데이터베이스의 구조와 제약 조건에 대한 명세를 기술한 메타데이터의 집합
데이터 사전에 저장되며, 인스턴스에 의해 규정됨
메타데이터: 데이터를 설명하기 위한 데이터, 데이터의 구조를 설명하고 검색에 활용
데이터 사전: 데이터베이스에 사용되는 데이터 구조 및 메타데이터를 저장하고 관리
인스턴스: 데이터 개체를 구성하는 속성에 대한 데이터 타입과 데이터 값

스키마의 3계층
외부 스키마(서브 스키마, 사용자 뷰)
사용자나 응용 프로그래머의 입장에서 필요한 데이터베이스의 논리적 구조를 정의, 여러 개의 외부스키마가 존재
개념 스키마(전체적인 뷰)
데이터베이스의 전체적인 논리적 구조, 개체 간의 관계와 제약조건, 접근 권한, 보안 및 무결성 규칙 명세를 정의, 데이터베이스 관리자에 의해서 구성
내부 스키마(저장 스키마)
물리적인 저장장치 입장에서의 구조

논리적 독립성(외부-개념 사이)
응용 프로그램에 영향을 주지 않고 논리적 구조 변경이 가능
물리적 독립성(개념-내부 사이)
응용 프로그램에 영향을 주지 않고 물리적 구조 변경이 가능

데이터베이스 사용자
DBA(DataBase Administrator)
데이터베이스 시스템의 모든 관리와 운영에 대한 책임을 지고 있는 사람이나 그룹
사용자 계정 및 권한 관리 / 성능 튜닝 / 구조 설계 및 유지 보수 / 장애 발생 시 복구 작업
응용프로그래머(Application Programmer)
DB를 사용하는 응용 프로그램을 개발하는 사람들로, SQL을 활용해 데이터를 조작
데이터 입력, 수정, 삭제, 조회 기능 구현 / UI 및 백엔드 작성 / SQL 쿼리 작성 및 최적화
일반 사용자(End User)
응용프로그램을 통해 DB에 접근해 정보를 검색하거나 입력하는 사용자
SQL 등의 지식 없어도 프로그램을 통해 DB 사용 가능
제한된 권한만 부여(읽기, 쓰기 등)

1980년대 기업내부 데이터베이스
OLTP(On-Line Transaction Processing)
다량의 온라인 단기거래 관리
정규화된 데이터, 소량의 데이터 연산, 실시간 데이터 수정, 조회, 주문입력시스템, 재고관리시스템 등에 활용
OLAP(On-Line Analytical Processing)
전문가의 데이터 분석에 초점
중복 데이터 저장 가능, 다차원 데이터 빠르게 접근, 대화식으로 의사결정 활용 정보 파악, 제품 판매 추이, 회계 분석 등에 활용

2000년대 이후 기업내부 데이터베이스
제조분야
CRM(Customer Relationship Management)
기업이 고객과 관련된 자료를 통합, 분석하여 수익을 창출하는 고객 관계 관리체계
SCM(Supply Chain Management)
제조, 물류, 유통 등 모든 유통 공급망에서 참여하는 업체 들과의 실시간 정보공유
생산, 유통 등의 모든 공급망 단계를 최적화하여 최적의 만족도 높은 제품을 제공
ERP(Enterprise Resource Planning)
인사, 재무, 생산, 판매, 경영 등 독립적 경영자원 시스템을 하나의 통합 시스템으로 구축
RTP(Real-Time Enterprise)
회사 내 전 부문의 정보를 하나로 통합하여 최신의 정보로 빠른 의사결정을 지원

금융분야
EAI(Enterprise Application Integration)
기업 내 ERP. CRM, SCM 등의 정보를 중앙 집중적 통합 솔루션
EDW(Enterprise Data Warehouse)
기존 Data Warehouse를 전사적으로 확대 및 재설계
블록체인
네트워크에 참여하는 모든 사용자의 거래내역 등의 정보를 분산 저장

유통분야
KMS(Knowledge Management System)
조직의 지식을 체계적으로 관리하는 시스템
RFID(Radio Frequency IDentification)
재고 현황을 실시간으로 파악
CRM, SCM

모델링
현실 세계의 정보를 약속된 표기법을 활용하여 데이터베이스 구조로 표현하는 과정
특징
단순화 – 누구나 쉽게 이해가 가능
추상화 – 주요 특징을 간략히 표현
명확화 – 모호하지 않은 표현

데이터베이스의 설계
개념적 설계
현실 세계에 대한 인식을 추상적 개념으로 표현하는 과정
E-R다이어그램을 작성
논리적 설계
물리적 장치에 저장할 수 있도록 컴퓨터가 이해 가능한 논리적 자료 구조로 변환
DBMS에 따라 서로 다른 논리적 스키마를 설계
물리적 설계
물리적 저장장치에 저장할 수 있는 물리적 구조의 데이터로 변환하는 과정
반응시간, 공간 활용도, 트랜잭션 처리량을 고려

E-R 다이어그램
개념적 데이터 모델의 가장 대표적인 예로서, 1976년 피터 첸에 의해 제안됨
개체(entity), 관계(relationship), 속성(attribute) 간의 관계를 시각적으로 묘사
데이터와의 관계를 쉽게 파악하며, 중복 제거 및 구조 최적화에 도움

개체와 속성
개체(학생)
사람이 생각하는 개념이나 정보 단위 같은 현실 세계의 대상체
유일한 식별자(학번)를 보유하고 있으며, 여러 속성으로 구성됨
속성(학번, 생년월일, 이름)
데이터의 가장 작은 논리적 단위로서 개체를 구성하는 항목
속성은 하나의 속성값을 가짐

관계
개체 간의 논리적인 연관성을 표현
구성 요소
차수 – 개체 간의 참여자 수를 표현(1:1, 1:M, M:N)
1:1            1:M           M:N
학생 – 학과.   학생 – 수강     학생 – 과목
관계명
관계의 이름(소속된다, 수강한다)
선택사양
개체가 항상 참여하는지 여부
