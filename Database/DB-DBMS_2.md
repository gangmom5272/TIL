관계형 데이터베이스 구성요소

테이블(개체, 릴레이션)
데이터베이스를 구조화하여 저장하는 단위
행과 열로 구성된 2차원 구조

속성(필드, 변수)
데이터베이스를 구성하는 가장 작은 논리적인 단위
각 속성은 고유한 이름을 가지며, 특성을 기술
속성의 수를 디그리(degree)라고 함

튜플(레코드)
테이블을 구성하는 각각의 행
속성의 모임으로 구성
튜플의 수는 카디널리티(cardinality)라고 함

테이블의 주요 특성
테이블의 유일성 – 중복된 튜플을 포함할 수 없다
튜플의 무순서성 – 행의 순서는 의미를 갖지 않는다
속성의 무순서성 – 속성의 순서는 중요하지 않다
속성의 원자성 – 각 셀에는 더 이상 나눌 수 없는 원자 값을 가져야 한다
속성명 고유성 – 속성명(컬럼명)은 중복될 수 없다

관계형 데이터베이스의 제약조건
키(Key) – 튜플들을 서로 구분할 수 있는 기준이 되는 속성 또는 속성들의 조합
키의 종류 – 후보키, 기본키, 대체키, 슈퍼키, 외래키

후보키
튜플을 유일하게 식별하기 위해 사용되는 속성 또는 속성들의 조합
모든 튜플에 대하여 유일성과 최소성을 만족해야 함
유일성: 하나의 튜플을 유일하게 식별 / 최소성: 튜플을 유일하게 식별하는 최소의 속성으로 구성
예시1: 학번, 주민번호, 이름이라는 속성을 갖는 학생 테이블이 있다고 가정
여기서 튜플을 유일하게 구분할 수 있는 속성은 학번, 주민번호가 된다. 이때 각 속성은 하나만 있어도 튜플을 구분할 수 있기 때문에 학번, 주민번호는 후보키가 된다.
예시2: 학번, 과목명, 학점이라는 속성을 갖는 수강 테이블이 있다고 가정
여기서 학번+과목명을 해야지 튜플을 구별할 수 있다. 학번만 불러서 학점을 묻는다면 어떤 과목인지 구분할 수 없고, 과목명만 불러서 학점을 묻는다면, 어떤 학생의 점수를 묻는지 알 수 없다. 이처럼 두 속성이 결합해서 후보키가 되기도 한다.

기본키
후보키 중에서 선택되어진 주 키(Main Key)
특정 튜플을 유일하게 구분할 수 있는 속성
NULL 값이나 테이블 내 중복된 값을 저장할 수 없다 -> 개체 무결성을 의미함
예시: 학생테이블에서 학번 또는 주민번호가 기본키로 설정 가능 / 수강테이블에서 학번+과목명을 기본키로 설정 가능
학생테이블에서 학번을 기본키로 설정하면 1,2,3,4,등의 값이 다른 튜플에 들어갈 수 없고 학번이 없는 학생은 존재할 수 없음

대체키
기본키를 제외한 나머지 후보키
예시: 학생테이블에서 학번을 기본키로 설정하면 주민번호는 대체키가 됨

슈퍼키
모든 튜플에 대해 유일성은 만족하지만 최소성은 만족시키지 못하는 키
예시: 학생테이블에서 학번, 주민번호, 학번+주민번호, 학번+이름, 학번+주민번호+이름 등으로 슈퍼키를 구성할 수 있음

외래키
한 테이블에서 다른 테이블의 기본키를 참조하는 키
테이블간의 관계를 정의하고 일관성을 유지에 사용
참조할 수 없는 외래키 값을 가질 수 없음 -> 참조 무결성
예시: 수강테이블이 학생테이블을 참조하고 있으며, 학생 테이블의 기본키는 학번이고 수강테이블의 학번은 외래키
수강테이블의 학번에는 학생테이블에 없는 학번이 들어갈 수 없다

정규화
이상현상
불필요한 데이터 중복으로 인해 테이블에서 삽입, 수정, 삭제 연산 시 발생할 수 있는 문제

삽입 이상
새로운 데이터를 삽입하려고 할 때, 다른 불필요한 데이터도 함께 입력해야하는 현상
예시: 학번,이름,과목명,교수명을 속성으로 갖는 테이블이 있다고 가정
아직 수강 신청을 안해서 과목명과 교수명을 기입할 수 없는 학생이 있다고 생각해보면 학번과 이름은 기입하고 과목명과 교수명은 NULL로 대체해야 함 

삭제 이상
데이터를 삭제할 때, 원하지 않는 다른 정보까지 같이 삭제되는 현상
예시: 휴학이나 자퇴로 학생의 학번과 이름을 삭제해야하는데 교수명과 과목명까지 같이 삭제해야하는 현상

갱신이상
중복된 데이터 중 하나만 변경하여 데이터 불일치가 발생하는 현상
예시: 수학을 2명의 학생이 수강중이라고 가정했을때, 교수가 바뀌어서 한 학생의 교수는 바뀐 교수명이 나오고 다른 수학을 수강중인 학생의 튜플에는 이전 교수의 이름이 기입되어 있는 현상

정규화
이런 이상현상을 해결하기 위한 과정
예시: 학생(학번, 이름), 수강(학번, 과목명), 과목(과목명, 교수명)으로 테이블을 나누면 됨

함수적 종속
결정자 A 속성에 의해 종속자 B 속성이 유일하게 결정되는 관계

완전 함수적 종속 – 종속자가 기본키의 모든 속성에 종속적인 경우
예시: 학번, 과목명, 교수, 시험점수라는 속성이 있는 테이블이 있다고 가정
이 테이블의 학번+과목명이 기본키로 설정됨 시험점수라는 값을 이 학번과 과목명 모두를 알아야 연결시킬 수 있음 학번만 알고 시험점수를 묻는다면 어떤 과목의 점수인지를 알 수 없고 과목만 알고 시험점수를 묻는다면 누구의 시험점수인지를 알 수 없음

부분 함수적 종속: 종속자가 기본키의 일부 속성에 종속적인 경우
예시: 교수는 과목명만 알아도 알 수 있음 이처럼 기본키의 부분만으로 연결 시킬 수 있으면 부분 함수적 종속임

정규화(1, 2, 3, BCNF, 4, 5) 
제 1정규화
모든 도메인이 원자 값(더 이상 쪼갤 수 없는 값)이어야 한다
예시: 학번, 이름, 과목명이라는 속성을 갖는 테이블이 있고 학번은 1 이름은 홍길동 과목명에는 데이터베이스,인공지능이 입력되어 있다고 가정
이는 제 1정규화를 위반함 왜냐하면 데이터베이스, 인공지능이 같은 튜플에 들어 있기 때문

제 2정규화 / 제 3정규화
부분 함수적 종속을 제거하여 기본키 전체에 종속되도록 한다(2)
이행적 함수 종속을 제거하여 비기본키가 다른 비기본키에 종속되지 않도록 한다(3)
예시: 학번, 과목명, 교수, 시험점수, 학점이라는 속성을 갖는 수강테이블이 있다고 가정하면 학번+과목명이 기본키가 됨 근데 이때 교수는 학번과 상관없이 과목명에만 종속되기 때문에 이걸 제거해줘야 함 
그리고 이 학교는 출석이런거 상관없이 시험점수만이 학점을 결정한다고 가정하면 학점은 시험점수에만 종속됨 즉 비기본키가 다른 비기본키에 종속되는 상황임
이처럼 2,3 정규화를 해결하기 위해서는 결정자는 남기고 종속자를 제거하면 됨
그래서 학번+과목명+시험점수를 묶어서 수강테이블로 만들고 과목명+교수를 묶어서 과목테이블, 시험점수+학점을 묶어서 성적테이블로 만들면 됨

BCNF
결정자가 후보키가 아닌 함수적 종속을 제거한다.
예시:
수업번호	교수	과목명	강의실
1	김교수	데이터	101
2	이교수	네트워크	102
3	박교수	인공지능	103
4	박교수	컴퓨터구조	103
이런 테이블이 있을때 기본키는 수업번호임 긜고 교수+과목명은 후보키임 그래서 교수 + 과목명을 알면 강의실을 알 수 있음 하지만 교수만 알아도 강의실을 알 수 있는 구조임 그래서 이걸 제거해주는게 BCNF임
그러면 교수+강의실로 테이블하나 만들고 수업번호+교수+과목명을 묶어서 테이블 만들면 됨 ->BCNF

제 4 정규화
다치 종속을 제거한다.
예시:
교수	과목명	언어
김교수	데이터베이스	한국어
김교수	데이터베이스	영어
김교수	운영체제	한국어
김교수	운영체제	영어
김교수	자료구조	영어
박교수	네트워크	영어
박교수	인공지능	영어
이런 테이블이 있다고 가정해보면 교수랑 과목명이 1:n관계이고 교수랑 언어가 1:n관계임 이거를 분리해주는게 4정규화임
그래서 교수+과목명 / 교수+언어 이렇게 테이블을 분리하면 됨

제 5 정규화
조인 종속을 제거하여 무손실 분해를 보장한다
예시: 위 4정규화와 같은 테이블이 있다고 가정했을 때, 김교수의 자료구조 수업에는 영어밖에 없지만 조인을 하게되면 김교수-자료구조-한국어 이렇게 매치되는 튜플이 생김 이걸 해결하려면 과목명+언어라는 데이블을 만들면됨

하지만 실무에서는 BCNF,4,5정규화까지는 잘 안함

반정규화
정규화는 데이터 중복 제거와 무결성 보장이라는 장점이 있지만, 현업의 실무 시스템에서는 여러 문제점이 존재
정규화의 문제점
조인과다, 복잡한 쿼리, 성능 저하, 불필요한 수준의 분리
그래서 이걸 해결하기 위해 반정규화를 함
반정규화란 정규화된 테이블을 의도적으로 다시 통합하거나 중복을 허용하여 성능을 높이는 작업

SQL의 개념
국제표준 데이터베이스 언어이며, 관계형데이터베이스(RDB)를 지원하는 언어로 채택
관계대수와 관계해석을 기초로한 데이터 언어

데이터 정의어(DDL)
논리적 데이터 구조와 물리적 데이터 구조의 사상을 정의 -> 데이터베이스 관리자나 데이터베이스 설계자가 주로 사용
데이터 조작어(DML)
응용 프로그램이나 질의어를 통하여 데이터를 실질적으로 처리 -> 데이터베이스 사용자가 주로 사용
데이터 제어어(DCL)
데이터의 보안, 무결성, 회복 등을 정의하는데 사용되는 언어 -> 데이터베이스 관리자가 주로 사용

DDL
CREATE – 스키마, 도메인, 테이블,  뷰 테이블, 인덱스 등을 정의하는 명령문
ALTER TABLE – 테이블의 정의를 변경하는 명령문
DROP – 스키마, 도메인, 테이블, 뷰 테이블, 인덱스 등을 삭제하는 명령문

DCL
GRANT: 데이터베이스 사용자에게 권한을 부여
REVOK: 데이터베이스 사용자에게 권한을 회수
COMMIT: 명령에 수행된 결과를 데이터베이스에 영구적으로 반영
ROLLBACK: COMMIT 전 변경사항을 되돌림
SAVEPOINT: 특정 지점을 지정하여 체크포인트를 설정
COMMIT, ROLLBACK, SAVEPOINT 는 TCL(트랜잭션 제어어)로 따로 구분하기도 한다

DML
SELECT문 – 데이터를 조회할 때 사용하는 가장 기본적인 명령어
SELECT – FROM – WHERE – GROUP BY – HAVING – ORDER BY 순으로 작성
FROM – WHERE – GROUP BY – HAVING – SELECT – ORDER BY 순으로 작동

WHERE절 – 특정 조건에 부합하는 데이터 조회할 때 사용
GROUP BY 절 – 특정 컬럼을 기준으로 행들을 그룹화하여 집계함수 결과를 산출
HAVING절 – 그룹화된 결과에 대한 조건
ORDER BY 절 – 결과를 정렬할 때 사용
